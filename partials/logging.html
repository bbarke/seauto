Logging

Logging is an essential part of an automation framework. If there is a failure, the exact cause must be found. SeAuto is build with slf4j, so implementing this logging will be easily done.

To add logging to any class file, simply add this to the top of the page: 


import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

public class Foo {

  private static final Logger LOG = LoggerFactory.getLogger(Foo.class);
  
  ...
}
Replace Foo for the actual class name you want to implement the logger. That is it! Your class is ready to start logging.

For every sample project, there is a log4j.properties located in src/test/resources folder. The format and logging locations can be changed by editing this file.

  • JBehave
  • Cucumber
  • JUnit

The log file, by default, is located at target/jbehave/_storiesLog.txt. It is located in this location, so the log file will be displayed in the top of the generated reports.

If you have read Running tests in Parallel then you understand that each thread is executed by a completely separate JVM. The regular threading model won't work. Simply logging what thread the current log statement is coming from will result in every test running in parallel reporting from the main thread. This is a major problem when trying to pinpoint a failure.

If running the tests with @RunWith(SeAutoCucumber.class), the name of the class that is executing the tests will be placed into the logger by default. In fact, that is currently the entire purpose of the SeAutoCucumber class.

If you are using the cucumber example project, then a log file is created and located in the target/seauto/tests.log

In addition to just logging, this can also help see what junit runner needs to be adjusted

Adding to this log file is just like any other framework.
